0.2 🔍 遗漏的风险与不足

🚨 遗漏一：Hermes 速率限制 + 高波动日并发瓶颈

你的报告假设 Hermes 随时可用，但：

  • DeepSeek API 有速率限制： 免费/低价 tier 通常 30-60 RPM（每分钟请求数）。高波动日如果有 20+ 条关键新闻需要分析，可能触发限流。
  • 延迟叠加： 每条 3-5 秒，5 条串行就是 15-25 秒。如果盘前报告要在 9:15 前出结果，必须提前至少 60 秒开始新闻分析。
  • API 不可用的日子： 网络故障、余额不足、DeepSeek 服务降级——你的报告提到了降级到词典，但没有量化降级后的准确率损失（词典 65% vs Hermes 85%，差 20 个百分点）。

建议：

  • 加入 OpenRouter fallback：在 Hermes 不可用时自动切到 Claude/GPT 做情感分析，多一条命
  • 盘前报告在 8:55 就开始预分析，留足 buffer
  • 对词典降级结果加 "confidence": "degraded" 标记，下游模型可以据此调低情绪维度的权重
⚠️ 遗漏二：A股新闻语境的特殊性，词典覆盖可能不够

通用情感词典（Hownet/NTUSD/清华）对以下 A 股特有表达几乎无效：

  • 反讽/黑话： "利好出尽是利空"、"老乡别走"、"韭菜收割机"——词典会把"利好"打正分，但实际含义是看跌
  • 谐音/代号： "宁王"（宁德时代）、"光伏茅"（隆基）——分词器拆成"宁/王"后失去语义
  • 政策用语： "精准有力" vs "合理充裕" 之间的微妙差异，词典无法区分

建议：

  • 在词典基础上加一个 A股专用补充词表（~500 词），收录常见黑话/谐音/政策用语
  • 把词典层定位为 "快筛"而非"判官"：只做初筛（|score|>0.5 才放过），其余全部交给 Hermes
  • Hermes 的 prompt 中加入 A 股语境提示："注意区分字面利好与实际利好，考虑政策用语的微妙差异"
⚠️ 遗漏三："中性陷阱"的 NLP 版本——你没提，但你知道它

十三哥，你们 6/9、6/11、6/12 连续三天踩的"中性陷阱"——模型输出全部"中性"，实际极差 14%+——这个问题在 NLP 情绪分析中 也会出现，而且更隐蔽。

具体表现：

  • 新闻标题通常温和中性（"A股三大指数微幅波动"），但正文可能暗藏关键信号
  • 大模型面对模糊文本时倾向于输出 score: 0.05, sentiment: "neutral"——和你们的融合评分"48-51"是同一个病根
  • 批量新闻都在 ±0.1 范围内打转，综合后给出"中性"结论，但市场实际大涨/大跌

建议（直接借用已有经验）：

  • 当 Hermes 连续 ≥3 条输出 |score| < 0.15 时，触发 "低信号预警"，不输出"中性"而是降级置信度
  • 在 prompt 中加一句："如果新闻无明显倾向，请给出'偏多'或'偏空'的倾向性判断而非中性，并标注低置信度"
  • 和你们的"北向连续+30亿→情绪上修"规则一样：给情绪模块设计一个 "累计偏差触发器"，连续 3 天词典+Hermes 都中性但大盘涨/跌 >1%，自动上修未来情绪偏置
💡 遗漏四：6维→7维的权重重分配需要更谨慎

你的报告建议 RS 27% + MOM 18% + 卡尔曼 14% + BBW 14% + 多因子 9% + 历史相似度 9% + 情绪 9%。这个分配方式是从各维等比例"切"出 9% 给情绪，没有考虑：

  • 情绪与哪些维度高度相关： 如果情绪和动量（MOM）高度共线（事实上它们经常共线——正面新闻→价格上涨→动量增强），那给情绪 9% + MOM 18% 实际是给同一信号 27%——过度加权
  • 情绪信号的噪声水平： 词典 65% 准确率意味着 35% 的时候它在输出噪声。直接给 9% 权重可能过高

建议：

  • Phase 3 上线后，先跑 2 周 shadow mode（情绪维度计算但不参与最终评分），收集数据后做相关性分析和回测，再决定精确权重
  • 初期权重保守：情绪 5-6%，从 RS 和 MOM 中各拿 2-3%，不要动卡尔曼和 BBW（它们是结构性的，和情绪不共线）

0.3 🚀 初一补充的更好方案

🌟 补充方案一：多模型仲裁（Multi-LLM Voting）

当 Hermes（DeepSeek）分析某条关键新闻时，如果同时调用一个更便宜/更快的模型做二次验证，可以大幅降低误判率：

  • 主裁判： DeepSeek Flash（¥0.001/次，3-5s）
  • 验票员： 同一 prompt 发给另一个轻量模型（或同一模型用不同 temperature），结果不一致时标记 confidence: "disputed"
  • 成本几乎不变： 只对 |score| > 0.5 的关键新闻做双验，每天可能只有 1-3 条

这个方案对于你们已经遇到过的"中性陷阱"特别有用——两个模型都输出"中性"的概率远低于单个模型。

🌟 补充方案二：来源权重（Source Weighting）

不是所有新闻来源的权重应该相同。一个简单的加权：

来源权重理由
证监会/央行/国务院公告2.0x政策直接信号，市场反应确定
公司公告（财报/重大事项）1.5x个股直接影响
券商研报/分析师点评1.0x专业但可能有利益冲突
财经媒体（东财/新浪/雪球）0.8x信息量大但噪音多
社交媒体（微博/股吧）0.3x散户情绪参考，噪音极高

在 sentiment_orch.py 中把来源权重乘入最终 score，可以让政策信号不被淹没在大量媒体噪音中。

🌟 补充方案三：情绪时间衰减（Sentiment Half-life）

新闻情绪不是永久的。一条利好新闻在发布后：

  • 发布后 0-2h： 权重 100%（市场正在消化）
  • 2-6h： 权重 70%（已被定价一部分）
  • 6-24h： 权重 30%（大多已反映在价格中）
  • >24h： 权重 10%（仅保留长尾效应）

实现很简单：final_score = raw_score * exp(-λ * hours_since_publish)，λ 约 0.05。这防止了旧新闻的"情绪尸体"污染当天的分析。

2. 方案详解

2.1 Hermes 分析方案 推荐

原理： 通过 CLI 调用初一（Hermes agent），将新闻文本传给 DeepSeek v4 模型做情感分析，返回结构化 JSON 结果。

调用方式：

import subprocess, json

def analyze_news_via_hermes(news_text):
    prompt = f"""分析以下A股新闻的情绪倾向，输出JSON：
{{
  "sentiment": "positive/negative/neutral",
  "score": 0.75,     // -1.0 ~ +1.0
  "confidence": "high/medium/low",
  "reason": "简短理由"
}}
注意：如果新闻无明显倾向，给出"偏多"或"偏空"的倾向性判断而非中性。
新闻：{news_text}"""
    
    result = subprocess.run(
        ['hermes', 'chat', '-q', prompt, '-m', 'deepseek/deepseek-v4-flash'],
        capture_output=True, text=True, timeout=30
    )
    return json.loads(result.stdout)
维度评估
准确率85-90% — 大模型对文本的理解力远超传统NLP，能识别反讽、隐喻、政策微妙差异
延迟3-5秒/条（含网络），批处理10条约15秒；建议盘前提前60s启动分析
成本DeepSeek Flash ¥0.001/次，每天100条约¥0.1。高波动日可能30-50条，¥0.03-0.05
优势零依赖、零部署、零维护、准确率最高、天然理解A股语境
劣势需网络、有调用成本（虽极低）、API速率限制（~60 RPM）、服务不可用时需降级
💬 初一补充

我在 prompt 中加入了"如果无明显倾向，给出偏多/偏空而非中性"这句话——这是直接从你们已有的"中性陷阱"经验来的。不加这句的话，大模型面对模糊新闻会输出 score:0.05，和你们融合评分的48-51是同一个病根。

2.2 情感词典方案 推荐（基础层）

原理： 用金融情感词典（约10,000词）+ jieba 分词 + 程度副词/否定词规则，对新闻文本做实时打分。 
import jieba

# 金融情感词典（开源资源）
# 来源1: 知网Hownet情感词典（免费）
# 来源2: 清华大学李军中文褒贬义词典
# 来源3: 台湾大学NTUSD简体中文情感词典
# 来源4: A股专用补充词表（自建，~500词）← 初一建议新增

pos_words = {"上涨", "突破", "利好", "大增", "盈利", ...}   # ~5000词
neg_words = {"下跌", "暴雷", "利空", "亏损", "违约", ...}   # ~5000词
a_share_special = {"利好出尽":"利空", "韭菜":"利空", "宁王":"宁德时代+利好", ...}  # ← 新增
intensifiers = {"非常", "极其", "大幅", "严重", ...}
negators = {"不", "没有", "并非", "未", ...}

def dict_sentiment(text):
    # 先处理A股特有表达
    for pattern, replacement in a_share_special.items():
        text = text.replace(pattern, replacement)
    words = jieba.lcut(text)
    score = 0
    for i, w in enumerate(words):
        multiplier = 1
        if i > 0 and words[i-1] in negators:
            multiplier = -1
        if i > 0 and words[i-1] in intensifiers:
            multiplier *= 1.5
        if w in pos_words:
            score += 0.1 * multiplier
        elif w in neg_words:
            score -= 0.1 * multiplier
    return max(-1, min(1, score))
维度评估
准确率65-75% — 简单规则，无法理解反讽/语境。加入A股补充词表后约提升至70-78%
延迟毫秒级 — 纯本地计算，可全天候运行
成本免费
优势零成本、可离线、全天候运行、适合做基础层快速过滤
劣势准确率低、无法处理复杂语句、A股黑话覆盖率需要持续维护
💬 初一补充

词典方案的核心价值不在准确率而在 "永不宕机"。它是安全网——即使一切外部服务挂了，它还能跑。另外，词典层应该定位为"快筛"而非"判官"：|score| < 0.3 直接放过，|score| > 0.3 才触发 Hermes 增强层。这样能减少 70-80% 的 API 调用。

2.3 FinBERT 方案 远期备选

原理： 加载 HuggingFace 上的预训练金融 BERT 模型，对新闻做三分类（正面/负面/中性）。

关键说明：

  • 推理 vs 训练： 方案中提到的 ~800MB 是推理（加载模型+分析新闻）的峰值内存，不是训练
  • 服务器894MB够吗？ ⚠️ 非常勉强。 BERT Base 模型约440MB + 分词器/运行时约600-700MB，剩余~200MB给系统，可能触发 OOM。如果同时有其他进程，基本必挂
  • 推荐做法： 在本地 Mac（内存充足）跑推理，把结果同步到服务器，不在服务器上加载模型
  • 中文FinBERT现状： 真正的"中文金融BERT"模型极少且质量参差。现阶段 Hermes 的准确率（85-90%）已经超过大部分中文FinBERT（70-80%），短期内没有替换的动力
from transformers import AutoTokenizer, AutoModelForSequenceClassification
import torch

# 推荐的中文金融情感模型（供参考）
# 方案A: 英文FinBERT + 翻译 → 不推荐，翻译会丢失金融语境
# 方案B: bert-base-chinese + 自标注微调 → 推荐但需要标注数据
# 方案C: 寻找开源中文金融BERT → 目前可用选项有限

tokenizer = AutoTokenizer.from_pretrained("bert-base-chinese")
model = AutoModelForSequenceClassification.from_pretrained("bert-base-chinese", num_labels=3)
# ⚠️ 需要在下游金融数据上微调后才可用，不能直接用

def finbert_sentiment(text):
    inputs = tokenizer(text, return_tensors="pt", truncation=True, max_length=512)
    outputs = model(**inputs)
    probs = torch.nn.functional.softmax(outputs.logits, dim=-1)
    return probs[0].tolist()  # [positive, negative, neutral]
💬 初一补充

FinBERT 路线有一个被你们忽略的 正向用途：不是替代 Hermes，而是作为 离线备用。如果 DeepSeek API 挂了、余额用完了、或者网络断了，FinBERT 可以在本地提供 70-80% 准确率的兜底。但这需要先把模型跑通——优先级放低，Phase 5 再说。

2.4 Tushare API 辅助参考

Tushare Pro 有 news 接口可以获取新闻资讯，但不自带情感分析标签。可以作为数据源，把获取的新闻喂给方案1或方案2做分析。注意 Tushare 的新闻接口有频率限制（通常200次/分钟），够用但不宽裕。

2.5 百度AI / 阿里云 NLP 备选

百度AI开放平台和阿里云都有情感分析API，费用约 ¥0.002-0.005/次，准确率80-90%。但需要额外开通服务，数据要经过第三方。作为应急备用可以，但不建议作为主力——成本和 Hermes 接近但灵活性和可定制性远不如。

3. 推荐方案：分层架构

🎯 最终推荐：情感词典（基础层）+ Hermes（增强层）双层架构 + 来源权重 + 时间衰减

3.1 综合评分权重

场景词典权重Hermes权重触发条件
批量扫描（>10条）60%40%只选 |score|≥0.3 的前3条做Hermes分析
关键新闻（≤5条）20%80%全部做Hermes分析 + 来源权重倍数
Hermes不可用100%0%降级到纯词典，标注 confidence="degraded"
连续≥3条中性——低信号预警：不输出中性，降级置信度

4.2 集成到统一模型

在 stock_analysis.py 的 get_model_summary() 中新增维度：

def calc_news_sentiment():
    """
    新闻情绪评分 → 0-100
    
    当前6维权重：
    RS 30% + MOM 20% + 卡尔曼 15% + BBW 15% + 多因子 10% + 历史相似度 10%
    
    第7维加入后建议权重（初一修订版）：
    RS 28% + MOM 17% + 卡尔曼 15% + BBW 15% + 多因子 9% + 历史相似度 8% + 情绪 8%
    
    ⚡ 初期上线：情绪先 shadow mode 跑2周，不参与最终评分
    ⚡ 正式接入：从RS和MOM中拿权重，不动卡尔曼和BBW（结构性的，不共线）
    """
    # 1. 获取当日新闻
    news_list = fetch_today_news()  # 含来源标签
    
    # 2. 词典基础层（全天候）
    dict_score = dict_batch_sentiment(news_list)
    
    # 3. Hermes增强层（仅分析 |score|≥0.3 的关键新闻）
    important = [n for n in news_list if abs(dict_score[n['id']]) >= 0.3]
    hermes_score = hermes_analyze_top_news(important[:3])
    
    # 4. 加权综合 + 来源权重 + 时间衰减
    combined = dict_score * 0.3 + hermes_score * 0.7
    source_factor = get_source_weight(news_list)  # 按来源
    time_factor = get_time_decay(news_list)       # 按发布时间
    
    # 5. 映射到0-100
    return max(0, min(100, combined * source_factor * time_factor * 50 + 50))

6. 风险与注意事项（完整版）

风险影响概率缓解措施
DeepSeek API 速率限制 / 不可用 增强层失效，准确率降至65-78% 中 OpenRouter fallback → Claude/GPT；词典基础层兜底；标注 confidence="degraded"
新闻获取失败（Tushare限额/东财反爬） 无输入数据 中 缓存上日结果，标注"数据滞后"；多数据源冗余（Tushare + 东财 + 新浪）
服务器894MB跑FinBERT OOM 进程崩溃 高 不要在服务器上加载模型。 本地推理 + 结果同步到服务器
情感词典准确率低 + A股黑话误判 噪声信号，可能误导模型 中 A股补充词表持续维护；Hermes增强层过滤；|score|<0.3 不入增强层
连续"中性"输出（NLP版中性陷阱） 情绪维度无效，浪费算力 高 低信号预警触发器；Prompt中强制倾向性判断；累计偏差自动上修
情绪与动量共线导致过度加权 模型偏向"追涨杀跌" 中 Shadow mode 验证；权重保守（5-6%）；从RS/MOM拿权重而非均摊
高波动日批量API调用导致成本激增 日成本从¥0.1飙升到¥0.5+ 低 限流：每日最多50条Hermes分析；超出部分纯词典

7. 💭 初一的最终总结

十三哥，你的报告方向完全正确。分层架构 + 零成本起步 + 3天上线的路径，在你们现有的约束下（服务器894MB、已有Hermes基础设施、已有6维评分体系）是最优解。

我的核心补充只有三点：

  1. 把"中性陷阱"的经验植入情绪模块。 你们6/9-6/12三次踩坑不是白踩的——"不做中性输出"的规则可以直接写进prompt和编排器逻辑里。
  2. 别急着给情绪维度权重。 先shadow mode跑2周，收集数据，做相关性分析。情绪如果跟动量高度共线，给9%就是给同一信号27%——这是最大的隐蔽风险。
  3. 把降级路径走通再上线。 Hermes挂了→词典兜底→标注confidence="degraded"→下游模型自动调低情绪权重。这条链路比"正常情况下的准确率"更重要。
