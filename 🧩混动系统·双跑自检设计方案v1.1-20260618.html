1. 背景与决策

1.1 背景

  • 新旧流程已解耦合，目前双跑状态（先旧后新），预计长期化
  • 8001设计文档 §10 定义了 L1-L4 四级自检框架
  • 当前 daily_check.py 主要覆盖旧流程

1.2 讨论结论

议题结论
框架结构✅ 采纳 每L层内部同时分 check_legacy + check_new，不在层级上划一刀
L0开关✅ 采纳 配置文件 [checks] mode = dual | legacy_only | new_only（十三哥确认双跑长期化）
函数族组织✅ 采纳 daily_check.py 内分 _check_legacy() / _check_new() 独立函数族，run_all_checks() 聚合
演进模型位置✅ 采纳 自检报告中独立区块，不合并到管线检查
双跑对比看版✅ 采纳 双层架构：每日快照（自检内嵌入）+ 独立看版页（周一刷新）
告警路由分层🟡 暂不实现 待告警通道就绪后再加

1.3 设计原则（8001对齐）

  1. 数据闭环自证 — 每个产物在上游有溯源、在下游有消费、在生命周期终了有归档
  2. L2→L3→L4三级自检形成完整证据链：数据新鲜度→结构完整性→产出质量
  3. 断点降级 — 一条数据路径在自检链上断点，该路径对应的报告降级或中止，不得静默通过
  4. clean code 退出 — 旧流程退役时直接删除 _check_legacy() 函数族，不伤新流程

8.3 一键恢复方案

restore.sh [--legacy | --new | --full | --config] [--date YYYYMMDD] [--verify]

恢复流程:
1. 从 _backup/{legacy,new,full,config}/ 找到对应日期tar.gz
2. 解压到临时目录 /tmp/restore_{timestamp}
3. md5sum 校验所有文件
4. 覆盖对应目录（备份目标目录的当前状态到 /tmp/restore_before/）
5. 恢复 crontab（如果备份中包含crontab）
6. 运行 python daily_check.py --verify-only 做恢复验证
7. 输出恢复报告：哪些文件恢复成功、哪些md5不匹配、自检是否通过

一条命令完全恢复: bash restore.sh --full --date 20260618 --verify

9. 🔍 演进模型实际状态扫描

9.1 设计思路

不依赖手动标记，用三层证据链自动判断任务是否提前完成：

层级检测方式判定规则示例
① 静态产物文件存在 + mtime如果.pkl或关键输出文件存在，且mtime ≥ 截止日期 00:00 UTC → 标记"已就绪"ridge.pkl 存在 + mtime=2026-06-16 ≥ 截止 2026-06-25 → ✅ 提前10天
② 运行时报告中真实出现在L3/L4输出或渲染后的HTML中 grep 模型关键词（如"脊回归"、"ridge权重"），如果命中→模型已上线运行报告中出现"脊回归系数" → Ridge已上线运行
③ 登记task_registry.json 自动更新自检扫描后自动写入 completed_at 时间戳{"ridge_online": {"completed_at": "2026-06-16"}}

9.2 任务注册表格式

config/task_registry.json:
{
  "ridge_online": {
    "deadline": "2026-06-25",
    "detection_type": "pkl_mtime",      ← 静态：检查.pkl文件mtime
    "detection_path": "models/ridge.pkl",
    "report_keywords": ["脊回归", "ridge权重"],  ← 运行时：报告中出现即认定上线
    "actual_status": "auto",            ← auto=自动判断 | manual=人工覆盖
    "completed_at": "2026-06-16",       ← 上次自动扫描发现就绪的时间
    "last_scan": "2026-06-18 01:00"     ← 上次扫描时间
  },
  "hurst_online": {
    "deadline": "2026-06-20",
    "detection_type": "source_func",    ← 检测分析引擎中函数是否已实现
    "detection_path": "scripts/analysis_engine.py",
    "function_name": "calc_hurst",
    "completed_at": "2026-06-10",
    "last_scan": "2026-06-18 01:00"
  }
}

9.3 扫描流程

  1. 每日L1自检时，遍历 task_registry.json
  2. 根据 detection_type 执行不同检测逻辑（pkl_mtime / source_func / file_exists / report_grep）
  3. 如果发现任务较截止日期提前完成，自动更新 completed_at
  4. 不要改 cron 时间——完成状态和cron调度是两件事。cron时间在设计文档里写，task_registry 只记录"实际完成时间"供自检报告引用
  5. 自检报告中演进模型区自动显示：该任务的「计划截止日」vs「实际完成日」，提前/延迟的天数

初一: "cron时间留着手动调就行。自动扫描只管识别完成状态，把状态反馈到自检报告里，不碰调度。"
